# ADR-014：开始/结束状态的强制迁移约束

## 背景与问题
为了让运行循环具备确定的起点与终点，并降低配置错误导致的死循环/不可达/不可结束等风险，需要对系统保留状态 `initial/end` 的迁移进行强约束。

## 决策（强制约束）
1. 开始状态 `initial`：**只能出，不能入**
   - 禁止任何迁移 target 指向 `initial`
2. 开始状态 `initial` 的所有外发迁移，其 `event` 名称 **必须为** `"start"`
3. 开始状态 `initial`：**至少有 1 个外发迁移**
4. 结束状态 `end`：**只能进，不能出**
   - `end.transitions` 必须为空（或不存在）
5. 结束状态 `end`：**至少有 1 个入边**（至少存在一条迁移 target 指向 `end`）
6. 除 `initial/end` 之外的其他状态：**至少有 1 个入边且至少有 1 个出边**
   - 入边：至少存在一条迁移 target 指向该状态
   - 出边：该状态至少包含 1 条外发迁移
7. 运行期语义：`initial` 的 Task **创建即完成**，并通过系统调用执行一次 emitterRules 产出事件 `"start"`（启动进入后续状态；默认可用 `system/start/auto-start`）
8. 运行期语义：`end` 的 Task **创建即完成**，并驱动 WorkflowRun 结束（Completed）
9. `initial/end` **不允许配置 `conditions`**（必须为空或不存在）
10. 同一状态内，`transitions[*].event` **不允许重复**（便于按事件名快速定位唯一迁移）
11. 同一迁移内，`targets[*].state` **不允许重复**（避免 fan-out 重复激活同一目标状态）
12. 同一 target 内，`prefetchers[*]` **不允许重复**（避免重复执行同一 prefetcher）

## 重要意义（评估）
这些约束的核心价值是：**把工作流的“可启动、可推进、可收敛”变成可校验的结构事实**，从而显著降低运行期的不确定性与排障成本。

1. **`initial` 只能出不能入**
   - 意义：保证运行的“起点唯一且稳定”，避免出现回边进入 initial 造成的“二次启动/重复初始化语义”。
   - 若不约束：有环图中一旦回到 initial，会让“重新初始化参数/重复创建 initial Task/重复 start 行为”等语义无法一致定义。

2. **`initial` 的所有外发事件名必须为 `start`**
   - 意义：形成统一的启动契约：引擎与调用方只需理解一个启动事件 `start`，减少事件命名分歧。
   - 若不约束：不同工作流可能用不同启动事件名，导致运行层接口与配置理解割裂，增加接入与运维复杂度。

3. **`initial` 至少 1 个外发**
   - 意义：保证工作流从起点可推进；否则会出现“可创建但永远无法离开 initial”的死流程。
   - 若不约束：会出现合法配置但永远无法产生后续 Task 的运行，属于低级配置错误。

4. **`end` 只能进不能出**
   - 意义：保证到达 `end` 即终止，形成明确的收敛点；特别适配“有向有环图”，避免终态仍可继续扩展出边。
   - 若不约束：会出现“到达 end 还能继续走”的歧义，终止条件无法一致定义（Completed 何时成立）。

5. **`end` 至少 1 个入边**
   - 意义：保证 end 可达，避免“永远结束不了”的工作流（没有收敛点）。
   - 若不约束：只能依赖运行时的间接判定（例如无可运行 Task）来结束，容易与并行/忽略任务等机制冲突并产生误判。

6. **除 `initial/end` 外其他 state 必须至少 1 入 1 出**
   - 意义：避免“孤岛状态”与“死胡同状态”，让状态机拓扑至少满足“可达且可离开”的基本健壮性。
   - 若不约束：
     - 只有入无出：运行会卡在该状态（除非靠 conditions→Ignored→事件绕行，但那也应显式配置）
     - 只有出无入：该状态永远不可达，属于无效配置

7. **initial 任务创建即完成并触发 start（由 auto-start 规则产出）**
   - 意义：把“启动”从普通业务任务中剥离成系统行为，保证每条运行都以统一方式进入后续状态，且不引入额外的人/系统处理步骤。
   - 若不约束：initial 可能被当成普通任务门控或外部处理，导致启动路径不一致、甚至阻塞在 initial。

8. **end 任务创建即完成并结束运行**
   - 意义：将 end 作为严格终态锚点：到达即结束，避免终态还需要额外处理才能结束造成歧义。
   - 若不约束：可能出现“到达 end 但运行仍在 Running（因为还在等待 end 的任务处理）”，破坏终止判定的一致性。

9. **initial/end 不允许配置 conditions**
   - 意义：initial/end 属于系统保留的“启动/终止锚点”，其行为是固定的（创建即完成并触发/结束），若允许条件门控会导致启动/终止语义不确定，甚至出现“无法启动/无法结束”的反直觉配置。
   - 若不约束：initial/end 可能被 conditions 拦截而进入 Ignored 等路径，破坏统一的 start/end 契约。

## 落地建议（配置校验）
- 这些约束应在**配置保存/发布**阶段做静态校验，避免把结构性错误推迟到运行期。
- 与当前“仅保留 `state.conditions`（脚本）”的门控机制结合时，建议在校验中同时检查：
  - 有 `state.conditions` 的 state 必须存在一条 `transition.event="ignored"` 的迁移，并在编排上收敛到 join（见 ADR-006/ADR-013）。
