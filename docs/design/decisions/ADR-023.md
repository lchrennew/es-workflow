# ADR-023：Prefetcher 参数声明字段命名为 parameters

## 背景
Prefetcher 的脚本会在运行期写入/派生数据，当前版本约定其返回 `result: Map<String,String>` 并由引擎合并到运行期参数字典（`run.livingParameters` 或 `task.livingParameters`）。

为了便于：
- UI 展示与理解（该 prefetcher 会写哪些 key）
- 配置治理与冲突检测（多个 prefetcher 写同名 key 的提示）
- 代码审计与变更影响评估（无需阅读脚本即可判断影响面）

需要在 Prefetcher 配置中增加“写入参数 key 的显式声明”字段。

## 决策
在 `PrefetcherSpec` 中增加可选字段：
- `parameters: List<String>`

用于声明该 prefetcher **可能写入/产出的参数 key 列表**。

## 选择 parameters 作为命名的原因
1. **职责单一、语义明确**：该字段只表达“参数 key 声明”，不暗示其等同于 prefetcher 的全部输出能力。
2. **避免 output 语义歧义**：prefetcher 的写入目标通常是 livingParameters（run/task），并非 Task 的 outputParameters；命名为 `outputKeys` 容易误导为“任务输出”。
3. **便于未来扩展**：若未来 prefetcher 能产出除参数外的其他数据（例如结构化数据集/文件/工件），应通过新增字段承载（例如 `spec.outputs/spec.artifacts`），而不是扩义或复用 `parameters`。

## 约束与说明
- `parameters` 不参与运行语义；运行时实际写入以脚本返回的 `result` 为准。
- 建议校验：`parameters[*]` 必须满足参数命名规则（大写字母/数字/下划线等，按现有参数规则）。
