# ADR-005：引入 WorkflowTask，并用 State.conditions（脚本）进行任务门控

## 背景与问题
工作流运行期不仅需要“当前激活了哪些状态”，还需要承载：
- 每个状态对应的处理工作（可被领取/执行/完成/审计）
- 运行期参数的作用域隔离（Run 级别 vs 状态任务级别）
同时，你希望不强行固定单一的并行汇合语义（AND/OR），而是为不同场景保留空间。

## 决策
1. 在 `WorkflowRun` 上新增 `tasks: List<WorkflowTask>`，Task 作为值对象（VO）。
2. 每当某个状态被激活，创建一个新的 Task，初始状态为 `initialized`。
3. 在配置域为 `WorkflowState` 增加 `conditions`（脚本），用于“任务门控”：
   - `WorkflowState.conditions` 通过后，Task 从 `initialized` 进入 `in-progress`。
4. 每个 Task 也拥有 `inputParameters/livingParameters/outputParameters` 三段式参数（Task 级别作用域）。

## 决策原因
1. **把“状态”与“可执行工作”解耦**：状态是图上的节点；Task 是运行时可操作的工作单元，更贴近真实系统（人/系统处理）。
2. **支持并行与汇合的多种策略**：先用 Task 的存在与状态来表达“分支进度”，后续可用条件/参数来实现不同 join 语义，而不在模型早期锁死。
3. **作用域清晰**：Run 参数承载全局上下文；Task 参数承载该状态处理过程的数据，避免污染全局。

## 影响与后果（待细化）
- 需要定义：Task 状态集合（initialized/in-progress/completed...）及其转换事件。
- 需要定义：并行分叉/汇合时 Task 去重、复用与触发规则（你提出的“分叉ID在参数中”的方案将作为下一步细化内容）。
