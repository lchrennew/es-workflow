# ADR-021：在 WorkflowTask 上增加 source（parent + event）以增强可追溯性

## 背景与问题
运行期需要更好地回答：
- 这个 Task 是如何产生的？是被哪个前序 Task 推进出来的？
- 推进它产生的“触发事件名”是什么（例如 `passed` / `rejected` / `start` / `b_skipped`）？

如果只保留 `WorkflowEvent`，虽然能看到事件发生，但在排障与审计中仍需要额外关联才能回答“某个 Task 的直接前因”。

## 决策
在 `WorkflowTask` 上增加值类型字段 `source`：
- `source.parent`：前序任务 id（ObjectId）
- `source.name`：触发事件名（内部事件，即 `transition.event`）

## 决策原因
1. **排障友好**：无需跨集合推导即可直接定位 Task 的前因。
2. **审计清晰**：能够明确记录“谁触发了该任务”以及“触发的事件名”。
3. **与 fan-out 兼容**：一次事件可以激活多个 target state，每个新 Task 都可携带同一个 `source`，便于追溯扇出链路。

## 影响与后果（待细化）
- 对于 `initial` 的 Task，`source` 可为空或按实现约定填充。
- 若未来引擎实现“同一 state 任务去重并合并多来源”，需要再扩展为多来源结构（例如 `sources: List<...>`）。
