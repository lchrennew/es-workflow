# ADR-004：WorkflowRun 参数字典（input/living/output）的引入与分层

## 背景与问题
Workflow 是“配置声明”，而 WorkflowRun 是“某一次具体运行”。运行期必须承载与业务相关的数据，用于：
- 驱动状态机迁移（条件判断、跳过状态、并行激活等）
- 为 prefetcher/业务步骤提供读写上下文
- 对外输出最终结果

如果不引入参数模型，就会出现：
- 运行推进缺少统一的数据来源（条件判断无法被一致地评估）
- 运行过程数据与最终结果混在一起，生命周期不清晰，难以审计与对外契约化

## 决策
在 `WorkflowRun` 中引入 3 个字符串字典（`Map<String,String>`）：
1. `inputParameters`：启动时传入的**全部入参**
2. `livingParameters`：运行中的实时参数值
   - 由 `inputParameters` 初始化
   - 仅在“开始后、结束前”存在
3. `outputParameters`：运行结束后的参数值
   - 仅在结束后存在

## 决策原因
1. **连接配置与运行**：参数是“同一份 spec 在不同输入下产生不同路径与结果”的关键载体。
2. **生命周期隔离，语义清晰**
   - input：固化“启动事实”，便于复现与审计
   - living：承载过程上下文，明确只在运行期有效，避免结束后被误用
   - output：固化对外可依赖的最终结果，形成清晰的外部契约
3. **便于治理与扩展**
   - 条件、prefetcher、事件处理都统一围绕参数读写展开
   - 后续若引入类型系统/作用域/敏感字段脱敏等治理机制，有明确落点

## 影响与后果（待细化）
- 需要进一步定义：
  - `livingParameters` 与 `inputParameters` 同名 key 的冲突策略（覆盖/禁止/命名空间）
  - `outputParameters` 在不同终态下的约定（例如是否输出错误码/原因等；当前版本仅定义 Completed）
  - 参数 key 的命名规范与命名空间（避免不同模块冲突）
