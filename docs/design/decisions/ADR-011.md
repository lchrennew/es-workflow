# ADR-011：Emitter 作为独立配置领域（kind/name/metadata/spec + allowedActions）

## 背景与问题
Emitter 需要同时满足：
1) 可配置的触发策略（任一响应/收齐响应/全通过/阈值等）  
2) 面向复杂业务的可编程扩展（脚本化）  
且不希望把策略细节耦合进 Workflow 配置本体，导致模型膨胀与职责混乱。

## 决策
1. 将 Emitter 抽为独立业务领域的配置聚合：`WorkflowStateEmitter`（业务场景 emitter）。
2. 配置结构与 Workflow 一致：`kind/name/metadata/spec`。
3. kind 固定为：`workflow-transition-emitter`。
4. Workflow 配置中的 `WorkflowState.emitter` 作为引用标识，指向 `WorkflowStateEmitter.name`。
5. `spec` 支持：
   - `allowedActions: List<{action,title}>`：对外可执行操作清单（第三方可提交的 `response.action`）及显示名称
   - `allowedEvents: List<{name,title}>`：可能产出的内部事件清单（对应 `transition.event`）及显示名称
6. `metadata` 支持：
   - `forUserState: Boolean`：标识该 emitter 是否允许被“用户新增的 state”选择/引用（用于 UI 过滤与治理）

## 决策原因
1. **边界清晰**：Emitter 与 Workflow 各自演进，避免将策略细节塞进状态机定义。
2. **复用与治理**：同一 emitter 可被多个状态引用，便于统一调整与版本治理。
3. **操作清单标准化**：把“可执行操作”从运行期/第三方实现中抽离到配置域，形成统一的 UI 展示与校验依据。

## 影响与后果（待细化）
- 需要定义：脚本的输入上下文与输出契约（至少 task/response；是否包含 responses 汇总视图等）。
- 需要定义：脚本安全与资源限制（超时、沙箱、可访问API等）。
