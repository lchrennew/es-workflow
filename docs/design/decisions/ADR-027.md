# ADR-027：区分 WorkflowEvent（人可读审计）与 Webhook（外部系统通知）

## 背景与问题
在运行期既需要：
1) 让用户查看“工作流都经历了什么”（面向人类阅读的时间线/审计）；  
2) 在特定时机或内部事件发生时，把信息推送给外部系统做集成（面向机器消费的通知）。

若将二者混为同一概念，容易导致：
- 内部审计模型被迫稳定对外契约，增加演进成本；
- webhook payload 直接暴露内部结构/实现细节，外部难以理解且存在安全风险；
- 运行期记录与对外通知的失败语义混淆（记录必达 vs 通知可能失败）。

## 决策
明确区分两类概念：

### WorkflowEvent（人可读）
- 定位：运行期审计时间线，用于 UI 展示“发生了什么、为什么发生、由谁触发”。
- 关注：可读性、解释性、可追溯（允许包含 title/color/icon/source 等解释字段）。
- 语义：作为运行期事实记录的一部分持久化保存。

### Webhook（外部系统可读）
- 定位：在特定时机/内部事件发生时，向外部系统发送通知（集成用途）。
- 关注：稳定、可机器处理、最小必要信息、避免泄露内部实现细节。
- 语义：发送失败不影响 WorkflowEvent 的记录；通知可采用“至多一次”语义（当前约定）。

## 二者关系（建议）
- webhook payload **可以引用** workflowEvent：提供 `workflowEventId` 以及少量摘要字段，便于外部系统关联与排障。
- 不要求 webhook 逐条镜像所有 WorkflowEvent；是否发送由“触发点列表”决定（运行域中定义）。
