# ADR-016：参数 key 命名规则与 value 类型统一为字符串

## 背景与问题
运行期存在多处“键值对字典”承载数据（run/task 参数、event payload 等）。若参数 key 命名不统一，或 value 类型多样，会导致：
- 跨模块数据对接困难（同一含义出现多种 key 风格）
- 条件/脚本/调试时出现大量类型与解析分支
- 存储与审计难以标准化

## 决策
1. 参数 key 命名采用：**大写字母/数字 + 下划线 `_` 分隔**，且首字符不为数字  
   - 末字符不允许为 `_`
   - 参考正则（允许 1 个字符）：`^[A-Z](?:[A-Z0-9_]*[A-Z0-9])?$`
2. run/task 参数以及 `WorkflowEvent.payload` 的 value 类型统一为：**字符串（String）**

> 注：`WorkflowResponse.payload` 允许为 `Map<String,Object>`，用于直接承载结构化数据（例如 update-task 指令的数组/对象）。

## 决策原因
1. **降低心智负担**：统一命名风格，减少“同一字段多种写法”。
2. **便于跨语言/跨存储**：字符串是最通用的交换格式，避免类型系统差异带来的兼容问题。
3. **利于治理与审计**：可统一做 key 校验、脱敏、索引与日志输出。

## 影响与后果（待细化）
- 对于 run/task 参数与 event payload：复杂结构需要以字符串承载（例如 JSON 字符串），并约定固定 key。
