# ADR-020：引入 EmitterRule 领域模型以复用“判定规则”

## 背景与问题
我们期望 emitter 能够跨 workflow/state 复用，避免为每个 state 定制开发一份聚合脚本。  
但如果允许一个 state 配置多个 emitter，会引入 `allowedActions` 合并与冲突等复杂度。

因此需要将“业务场景（操作清单）”与“判定规则（触发内部事件）”解耦：
- Emitter：负责声明可执行操作清单（UI/校验/对外契约）
- Rule：负责在收到 response 后判定是否产出内部事件

## 决策
1. 引入独立配置实体 `EmitterRule`：
   - 配置结构：`kind/name/metadata/spec`
   - kind 固定为：`emitter-rule`
   - spec 仅包含 `script`（仅填写 content；运行时按统一函数签名包装）
2. WorkflowState 上采用组合方式：
   - `state.emitter`：引用一个 `WorkflowStateEmitter`（声明 allowedActions）
   - `state.emitterRules[*]`：有序引用多个 `EmitterRule`（规则链，短路）

## 决策原因
1. **规则复用**：一条规则可被多处引用，提升复用与治理效率。
2. **降低冲突**：避免多 emitter 带来的 allowedActions 合并/冲突问题。
3. **职责更清晰**：操作清单归 emitter 管；触发判定归 rule 管。

## 影响与后果（待细化）
- 需要定义 rule 的执行上下文与可访问数据（run/task/request/parameters/api）。
- 需要定义 state.emitterRules 为空时的行为（是否允许“不靠 response 推进”）。
