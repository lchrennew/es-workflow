# ADR-035：移除 RequestSender 配置域，统一使用 Webhook 机制

## 状态
已采纳（2026-05-05）

## 背景
在 ADR-025 中引入了 RequestSender 配置域，用于将 WorkflowRequest 投递到外部系统/通道。

同时，系统中还存在 Webhook 机制，用于在运行期关键时机向外部系统投递通知。

经过分析发现，RequestSender 和 Webhook 存在职责重叠：
1. **都是向外部系统发送消息**：RequestSender 发送请求，Webhook 发送通知
2. **实现机制相似**：都可能使用 HTTP POST、MQ、IM 等通道
3. **配置复杂度增加**：需要维护两套独立的配置领域

## 决策
移除 RequestSender 配置域，统一使用 Webhook 机制处理所有外部通信：

1. **删除 RequestSender 配置域**：
   - 删除 `17-request-sender-domain.md`
   - 从术语表中移除 RequestSender 相关术语

2. **Webhook 承担双重职责**：
   - **事件通知**：run.started、task.completed 等（fire-and-forget）
   - **请求发送**：通过 WorkflowRequest 机制发送需要响应的请求

3. **request.target 语义调整**：
   - 不再依赖 `<senderName>:<ref>` 格式映射到 RequestSender
   - 直接使用目标标识，如：
     - `user:<工号>`：用户审批
     - `sys:<系统标识>`：系统调用
     - `webhook:<url>`：Webhook 调用

4. **废弃 ADR-025**：标记为已废弃，由本 ADR 取代

## 决策原因
1. **简化架构**：减少一个配置领域，降低系统复杂度
2. **职责统一**：所有外部通信统一由 Webhook 机制处理
3. **避免重叠**：RequestSender 和 Webhook 的职责边界模糊，容易混淆
4. **实现简化**：不需要维护两套独立的发送机制

## 影响
- 删除 RequestSender 配置域及相关文档
- Webhook 机制需要支持 WorkflowRequest 的发送（不仅是事件通知）
- request.target 的解析逻辑简化，不再需要映射到 RequestSender.name
- ADR-025 标记为已废弃

## 实现建议
Webhook 机制可以根据 request.target 的前缀来决定发送方式：
- `user:` → 发送到用户审批系统
- `sys:` → 调用内部系统 API
- `webhook:` → 直接 HTTP POST 到指定 URL
- 其他自定义前缀 → 根据配置路由到相应的通道
