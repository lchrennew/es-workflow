# ADR-025：引入 RequestSender 配置域（request target 前缀映射 name）

## 背景与问题
当前工作流模型中：
- prefetcher 产出 `TMP_REQUEST_TARGETS`（逗号分隔的 `<prefix>:<ref>` 列表）
- Task 进入 `InProgress` 时生成 `WorkflowRequest` 列表（见 ADR-018）

但“请求生成之后如何投递到外部系统/通道”（HTTP/MQ/IM/内部系统等）缺少统一的领域建模，导致：
- 发送逻辑散落在实现中，难以治理与复用
- 难以做到配置即代码（可检索/可版本/可审计）

## 决策
引入独立配置领域模型 `RequestSender`，遵循统一的配置结构：
- `kind / name / metadata / spec`

其中：
- `kind` 固定为：`request-sender`
- `name` 与 request target 的前缀严格一一对应：
  - request target 表达式：`<senderName>:<ref>`
  - 强约束：`senderName == RequestSender.name`
- `spec.script` 提供发送逻辑（JavaScript，content 形式），用于把 `WorkflowRequest` 投递到外部系统/通道，并返回发送回执/状态信息。

## 运行期参与时机（关键）
1. 迁移激活某 state，创建 `WorkflowTask`
2. Task 进入 `InProgress`
3. 引擎根据 `TMP_REQUEST_TARGETS` 生成 `WorkflowRequest` 列表（见 ADR-018）
4. 对每条 `WorkflowRequest`：
   - 解析 `request.target` 得到 `<senderName>:<ref>`
   - 加载 `RequestSender(kind="request-sender", name=senderName)`
   - 执行 `sender.spec.script` 完成发送
5. 外部系统提交 `WorkflowResponse`
6. `state.emitterRules` 聚合 responses，产出内部事件，推进状态机

## 决策原因
1. **职责清晰**：prefetcher 负责“发给谁”（产生 targets）；RequestSender 负责“怎么发”（投递通道与协议）。
2. **配置即代码友好**：以 `kind+name` 为检索键/落盘路径，无需额外索引字段（不引入 `spec.type`）。
3. **可扩展**：新增一种发送通道仅需新增一个 RequestSender（新增 name 前缀），不影响 workflow 配置语义。

## 约束建议
- 若运行期出现 `senderName` 无对应 RequestSender 配置，应直接报错并拒绝发送（避免静默丢请求）。
