# ADR-003：条件从 when 演进为 conditions + prefetchers（已废弃）

> 状态：已废弃（历史记录保留用于追溯）  
> 替代方案：见 ADR-013（仅保留 state.conditions（脚本）用于任务门控），以及 Prefetcher 独立配置领域 ADR-019。

## 背景与问题
在目标状态激活前，需要：
1) 用“结构化条件”而非单字符串表达式来描述判断规则；  
2) 在条件判断之前支持“预取数据”（例如查询业务数据、读取上下文）以供判断使用。

## 决策
将 `WorkflowTransitionTarget` 从：
- `when: String?`
演进为：
- `conditions: List<Condition>`（可选，键值规则集；后续已演进为“conditions 为脚本 content”）
  - `key: String`
  - `op: String`
  - `value: String`
- `prefetchers: List<String>`（可选，仅声明预取器标识；预取器的具体配置在其他位置维护）

> 备注：由于后续选择“仅保留 State.conditions（脚本）”，`WorkflowTransitionTarget.conditions` 已移除，本 ADR 仅作为历史记录保留（见 ADR-013）。

## 决策原因
1. **结构化可治理**：`key/op/value` 更利于校验、静态分析与统一运算符集合管理。
2. **可支持“先取数再判断”**：`prefetchers` 明确表达依赖数据获取步骤，避免把“取数逻辑”塞进条件字符串。
3. **演进空间更大**：未来 Condition/Prefetcher 都可扩展更多字段，而不破坏既有概念边界。

## 影响与后果
- 需要定义：Condition 的 `key` 命名空间、可用 `op` 集合、以及 `value` 的类型系统（当前先用 String 占位）。
- 需要定义：prefetcher 的注册与配置存储方式（本模型中仅声明标识字符串）。
