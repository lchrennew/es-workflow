# ADR-007：在 Task 上引入外部 Request/Response

## 背景与问题
任务进入 `InProgress` 后，经常需要与外部处理方交互（人/系统），形成“发出请求 → 得到响应”的闭环。  
我们需要一个运行期结构来承载这些交互数据，用于路由、审计与后续状态推进（推进方式后续再定）。

## 决策
1. 在 `WorkflowTask` 上增加 `requests: List<WorkflowRequest>`。
2. `WorkflowRequest` 最多拥有一个 `response: WorkflowResponse`（0..1）。
3. `WorkflowRequest` 记录：
   - `runId`（来自 `WorkflowRun.id`）
   - `taskId`（关联的 Task 标识，ObjectId）
   - `target`（例如 `user:<工号>` / `sys:<系统标识>`）
   - `createdAt`
4. `WorkflowResponse` 记录：
   - `id`（ObjectId，可从中推导创建时间）
   - `action`（响应动作标识）
   - `kind`（操作类别：decision/update-task）
   - `payload`（响应数据）
5. 当前阶段不为 Request 单独建立状态机（用“是否已有 response”表达闭环即可）。

## 决策原因
1. **交互闭环可追溯**：把对外请求与响应沉淀为模型要素，方便审计与问题定位。
2. **保持低复杂度**：先不引入 Request 状态机，避免状态爆炸；后续需要超时/重试再扩展。
3. **路由与归因清晰**：`target` 以统一格式表达“由谁/哪个系统处理”，便于权限、通知与统计。
4. **action/payload 不放在 request**：第三方可通过 `runId + taskId` 拉取上下文，并从 **state.emitter 的 `actions`** 获取可执行操作清单（含 `kind`）；响应再用 `response.action` 表达选择结果。

## 影响与后果（待细化）
- 需要定义：response 到达后如何映射为工作流事件（fireEvent）或如何写入参数（living/output）。
- 需要定义：requestId 的幂等与去重策略（当前用 ObjectId 占位）。
