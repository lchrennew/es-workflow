# ADR-015：Task 终态时将 outputParameters 合并到 run.livingParameters

## 背景与问题
运行期需要把每个任务的处理结果沉淀为全局上下文，以便：
- 后续任务的条件判断、prefetcher、以及 emitter 脚本可读取此前结果
- 分支汇合（join）时基于多个任务的输出做聚合决策

如果 Task 的输出仅停留在 Task 内部，会导致跨状态的数据传递依赖额外机制，增加实现复杂度与排障成本。

## 决策
当任一 Task 进入终态（`Completed` / `Ignored`）时：
- 将 `task.outputParameters` 合并到 `run.livingParameters`

## 决策原因
1. **统一数据通道**：run.livingParameters 成为“运行中全局可读写上下文”，减少跨状态数据搬运。
2. **便于编排与汇合**：join/聚合逻辑只需读取 run.livingParameters 即可判断分支完成情况与结果。
3. **可观测性更强**：关键结果集中在 run.livingParameters，有利于追踪运行路径与问题排查。

## 影响与后果（待细化）
- 需要定义合并策略：同名 key 冲突时的覆盖规则（最后写入 wins / 禁止覆盖 / 命名空间）。
- 需要约定输出 key 的命名规范（避免不同任务互相覆盖）。
