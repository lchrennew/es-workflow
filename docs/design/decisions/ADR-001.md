# ADR-001：WorkflowTransition 的归属（方案A vs 方案B）

## 背景与问题
在 `Workflow.spec` 中，我们需要表达一个“以状态为节点、以事件触发迁移”的有向有环图。  
关键取舍是：**迁移（WorkflowTransition）应该归属到 WorkflowState 内，还是由 WorkflowSpec 统一维护？**

## 备选方案

### 方案A：WorkflowState 内含 transitions（邻接表 / 外发边）
- `WorkflowSpec.states[]`
  - `WorkflowState.transitions[]`
    - `event`
    - `toStates[]`

### 方案B：WorkflowSpec 集中维护 transitions（边表 / 全局边集合）
- `WorkflowSpec.states[]`
- `WorkflowSpec.transitions[]`
  - `fromState`
  - `event`
  - `toStates[]`

## 决策
选择 **方案A：WorkflowState 内含 transitions**。

## 决策原因
1. **语义贴合状态机**：迁移天然是“从某状态出发的外发边”，属于状态的组成部分，模型更直观。
2. **降低冗余与不一致风险**：不再需要 `fromState` 字段，减少“边表与状态集”之间的重复表达，降低写错/漏删导致的不一致概率。
3. **聚合边界更清晰**：编辑某状态时，其外发迁移随状态一起变更，符合 DDD 中“对象内部一致性由聚合维护”的思路。
4. **便于增量演进**：后续在状态上扩展属性（如 entry/exit 行为、是否终止态等）时，结构仍保持清晰。

## 影响与后果
- `spec` 的存储形态更偏向“邻接表”，全局按 `event` 检索迁移时需要遍历 `states`（如有性能诉求，可在运行时构建索引或在读取时做缓存）。
- 后续若迁移需要“一等对象”能力（例如独立 ID、审计、复用、被其他结构引用），可能需要重新评估是否引入边表或派生视图。
