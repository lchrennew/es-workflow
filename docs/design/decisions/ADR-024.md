# ADR-024：进入非 end 状态的迁移 target 必须配置“产出 TMP_REQUEST_TARGETS 的 prefetcher”

## 背景
当前版本的工作流推进模型以 **request/response** 为核心：
1) Task 进入 `InProgress` 时，若存在 `task.inputParameters["TMP_REQUEST_TARGETS"]`，引擎会据此生成 `task.requests`（见 ADR-018）。  
2) 外部系统对 requests 提交 `responses`。  
3) `state.emitterRules` 聚合 responses，产出内部事件（`transition.event`）。  
4) 内部事件驱动状态迁移，激活后续 states。

因此，对于“进入非 end 状态”的迁移 target，如果无法产生 request，则不会有 response，也就无法触发 emitter 产出事件，状态机无法向后流转（流程卡死）。

## 决策
在 Workflow 配置域增加约束：
- 对任一 `transition.targets[*]`，若 `target.state != "end"`，则 `target.prefetchers` 必须至少包含 1 个“会产出 `TMP_REQUEST_TARGETS` 参数”的 prefetcher。

判定方式：
- 该 prefetcher 的配置中 `spec.parameters` 包含 `TMP_REQUEST_TARGETS`（见 `12-prefetcher-domain.md`）。

## 决策原因
1. **保证可流转性（liveness）**：确保进入每个可处理状态前都能产生 requests，从而具备推进链路：`TMP_REQUEST_TARGETS → requests → responses → emitter event → transitions`。
2. **静态可校验**：把运行期隐式依赖（必须能派单）提升为配置契约，避免“能编排但运行卡死”的配置。
3. **提升编辑体验**：UI 可在边/target 级提示“缺少派单 prefetcher”，并支持一键修复（补充默认 prefetcher）。

## 影响与后果
- 该约束强化了“用户新增 state 依靠 request/response 推进”的统一模型；若未来引入无需 request 的自动节点/定时节点，应通过新增机制或例外规则放宽该约束。
