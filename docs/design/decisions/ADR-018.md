# ADR-018：基于 TMP_REQUEST_TARGETS 生成 Task 的 request 列表

## 背景与问题
我们希望 request 尽可能轻量：不在 request 中携带 action/payload，而由第三方通过 `runId + taskId` 获取上下文，并从 **state.emitter 的 `allowedActions`** 获取可执行操作清单。  
因此需要一个机制来回答：Task 进入 `InProgress` 后“要通知哪些处理方参与处理”。

## 决策
当 Task 进入 `InProgress` 时：
- 若 `task.inputParameters` 存在参数 `TMP_REQUEST_TARGETS`，其值为以英文逗号 `,` 分隔的 target 列表（字符串）
- 引擎按该列表为每个 target 创建一条 `WorkflowRequest(runId, taskId, target)`

## 决策原因
1. **低复杂度**：不新增额外的 request-planner 配置域，直接用参数表达“发给谁”。
2. **动态性**：target 列表可由上游任务输出（合并到 run.livingParameters）后再写入 task.input，实现动态分派。
3. **契合 TMP_ 语义**：该参数仅用于任务运行期派单，任务结束后不输出、不后传，避免污染全局上下文。

## 影响与后果（待细化）
- 需要定义字符串解析规则：是否 trim 空格、是否允许空项、target 去重策略等。
- 若缺少 `TMP_REQUEST_TARGETS`：Task 是否允许“无请求推进”（纯内部处理）需要在引擎层明确。
