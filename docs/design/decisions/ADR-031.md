# ADR-031：WorkflowTask 持有运行期 State 快照（name/emitter/emitterRules/transitions）

## 背景与问题
当前 Task 是由某个 `WorkflowState` 激活后创建的运行期对象。

如果 Task 运行过程中仍持续依赖配置域中的原始 `WorkflowState`，会带来几个问题：
1. 运行时语义分散：Task 自身不能完整表达"我该如何继续运行"。
2. 排障与审计不直观：查看某个 Task 时，仍需要回到配置域查找其 state 配置。
3. 难以支持运行期动态增添任务的场景（例如前/后加签）：新任务若不绑定某个静态 state，很难获得完整的运行所需字段。

## 决策
在 `WorkflowTask` 上直接保存运行期所需的 state 快照字段：
- `name`（原 `stateName` 更名为 `name`）
- `emitter`
- `emitterRules`
- `transitions`

**不快照 `conditions`**：
- `conditions` 只在 Task 创建时执行一次判断（Initialized → InProgress/Ignored）
- 判断完成后不再使用，快照无运行期价值
- 如需审计"为什么这个 task 被 ignored"，可通过 task.name 追溯到原始 state 配置

创建 Task 时，从对应 `WorkflowState` 复制上述字段到 Task。

## 决策原因
1. **运行期闭包更完整**：Task 自身即可描述其动作清单、规则链与外发迁移。
2. **降低后续依赖**：Task 创建后，后续运行尽量不再依赖配置域中的原始 `state`。
3. **支持动态任务**：便于支持前/后加签等运行期动态增添任务的场景。
4. **审计友好**：查看 Task 即可知道其运行时采用的规则快照。
5. **避免冗余**：conditions 只用一次，不快照可减少存储开销。

## 影响与后果
1. Task 从"仅记录状态名"提升为"运行期状态快照"。
2. `Initialized -> InProgress/Ignored` 的判定读取 `state.conditions`（从配置域），不快照到 task。
3. 事件推进时，已有 Task 的外发迁移读取 `task.transitions`；仅在创建新 Task 时，才从目标 `WorkflowState` 复制快照。
