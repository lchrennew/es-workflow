# ADR-022：在 WorkflowTask 上增加 endEvent 记录任务结束事件

## 背景与问题
运行期可视化与审计经常需要回答：
- 某个 Task 最终是如何结束的？（通过/拒绝/忽略等）
- 不希望每次都去 `run.events` 中做“最后一条事件”的二次推导与关联。

现有 `task.source` 解决“从哪来”，但缺少“怎么结束”的直接字段。

## 决策
在 `WorkflowTask` 上增加字段：
- `endEvent: String?`
- `endEventId: ObjectId?`（对应 `WorkflowEvent.id`）

## 赋值规则（约定）
- 当 Task 进入 `Ignored`：固定写入 `"ignored"`
- 当 Task 进入 `Completed`：写入导致该 Task 完成并推进的内部事件名（`transition.event`，例如 `"passed"` / `"rejected"` / `"start"`）
 - 同时写入 `endEventId`，用于精确关联导致任务结束的那条事件记录

## 影响与后果
1. **可视化友好**：Task 卡片/节点可直接展示终态与结束原因（事件）。
2. **排障更快**：无需额外查询/关联即可定位“最后推进事件”。
3. **可扩展**：如需更强追溯，可通过 `endEventId` 精确关联导致任务结束的那条 `WorkflowEvent`。
