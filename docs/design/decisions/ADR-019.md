# ADR-019：Prefetcher 作为独立配置领域（kind/name/metadata/spec + script）

## 背景与问题
工作流运行期存在“先取数/派生数据”的需求，用于：
- 任务门控（state.conditions，脚本）的判断所需上下文
- 任务处理与第三方展示上下文
- emitter 脚本判断的输入补全

如果把取数逻辑直接硬编码在引擎或散落在各处，会导致复用困难、治理困难，也不利于审计与演进。

## 决策
1. 将 Prefetcher 抽为独立业务领域的配置实体：`Prefetcher`。
2. 配置结构与 Workflow 一致：`kind/name/metadata/spec`。
3. kind 固定为：`prefetcher`。
4. `spec` 仅包含 `script`（脚本文本，运行时默认按 JavaScript 执行；且脚本仅填写 content，运行时按统一函数签名包装）。
5. Workflow 配置中的 `WorkflowTransitionTarget.prefetchers[*]` 作为引用标识，指向 `Prefetcher.name`。

## 决策原因
1. **职责解耦**：把“取数/派生”从核心引擎逻辑中剥离，保持引擎通用与稳定。
2. **复用与治理**：同一 prefetcher 可被多处引用，便于统一管理、版本治理与审计。
3. **可编程扩展**：以脚本提供足够表达能力，适配不同数据源与派生逻辑。

## 影响与后果（待细化）
- 需要定义脚本输入上下文（run/task/事件/payload 等）与输出写入位置（写 run.living 或 task.living）。
- 需要定义脚本安全与资源限制（超时、沙箱、可访问 API）。
