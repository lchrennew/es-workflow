# ADR-013：移除 TransitionTarget.conditions，仅保留 State.conditions（脚本）

## 背景与问题
同时在 `TransitionTarget` 与 `State` 上建模条件，会带来：
- 条件语义分裂（激活条件 vs 门控条件）导致理解成本高
- 文档与实现复杂度上升，且容易产生“条件到底应该放哪” 的不一致

## 决策
- 移除 `WorkflowTransitionTarget.conditions`
- 仅保留 `WorkflowState.conditions`（脚本），用于 Task 门控（Initialized → InProgress / Ignored）
- 如需“跳过/绕行”效果，通过 `state.conditions + ignored 事件迁移` 实现

## 沟通确认的 5 点（设计依据）
1. **分支必须被激活（一定创建 Task）**  
   为支持分支汇合（join）逻辑，需要区分“尚未完成”与“已完成但忽略（Ignored）”两种情况；因此不能用“未激活/未创建”来表达跳过。
2. **过程透明、易追踪**  
   通过创建 Task + Ignored 完成态，使“分支发生了什么”可观测、可排查；并为 join 提供可判断的事实依据。
3. **prefetchers 仍然优先于条件判断**  
   保持“先取数、后判断”的顺序：迁移时执行 `target.prefetchers` 获取数据；随后由 `state.conditions` 在任务门控阶段做判断。
4. **强制：有 conditions 必须配置不通过时的流转**  
   若 state 配置了 `conditions`，则必须提供“条件不通过时的状态流转”（`transition.event="ignored"`）并收敛到后续 join，避免悬挂分支。
5. **state 激活是逻辑视图，物理事实以 Task 为准**  
   “state 被激活”是逻辑概念；更物理/可持久化的状态是：该 state 下是否存在未结束的 Task（由 Task.status 反映）。

## 决策原因
1. **降低整体复杂度**：统一条件落点，减少分支语义。
2. **逻辑更清晰**：迁移只负责拓扑（event→targets），状态条件只负责任务可执行性。
3. **更贴合当前运行机制**：任务必然创建，再由条件决定是否处理或忽略，符合 Ignored 模型。

## 影响与后果（已确认 / 待细化）
- **强校验**：若 `WorkflowState.conditions` 非空，则必须存在 `transition.event="ignored"`，且该事件必须收敛到后续 join（汇合）状态。
- 需要定义：触发事件时使用的参数上下文（run/task 的 input/living/output 如何传递或合并）。
- 若未来需要更强的“结构化并行控制/严格汇合（join）统计”，可再评估引入分叉标识（ForkID）等机制；当前版本优先保持事件驱动的清晰与低复杂度。
