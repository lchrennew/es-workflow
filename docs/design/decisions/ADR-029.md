# ADR-029：Emitter actions 增加 kind（decision/update-task），并通过 update-task response.payload 支持增删/作废请求

## 背景与问题
当前工作流推进以 request/response 为核心：response 进入 emitterRules 产出内部事件，驱动状态迁移。

但存在一类操作仅用于“更新当前任务”，不应推进状态机，例如：
- 加签：为当前任务新增处理方（新增 requests）
- 删除请求：对某些 requests 执行“作废（void）但不产生替代 request”的留痕操作
- 撤回响应：对已响应 request 执行“作废（void）并创建替代 request 重新发起”的留痕操作

如果把这类操作与“推进流转”的 action 混在同一条 emitterRules 里，会导致 UI/校验/运行语义不清晰。

## 决策
1. Emitter 的 actions 统一收敛为一份清单：
   - `WorkflowStateEmitter.spec.actions: List<AllowedAction>`
   - 其中 `AllowedAction.kind ∈ {"decision","update-task"}` 用于区分决策类/更新任务类
2. 运行时路由规则：
   - 若 `action.kind="decision"`：执行 `state.emitterRules`，产出内部事件并迁移
   - 若 `action.kind="update-task"`：不产出内部事件、不迁移；引擎直接读取 `response.payload` 描述的更新意图并应用到当前 Task

## request-response 追加约束（新增，已确认）
1. 每个 request 可以对应多个 responses（response 为日志）。
2. responses 中最多只能有 1 条“决策类 action”的 response，且必须排在最后（出现决策类 response 后，该 request 不再接受后续 responses）。
3. emitter 的 action 需要区分“决策类/非决策类”：
   - `actions[*].kind="decision"`：决策类（参与 emitterRules 统计与事件触发）
   - `actions[*].kind="update-task"`：非决策类（仅更新 task，不参与事件触发）

## 可更新内容（当前已确认）
- 增加 request
- 作废 request（用于“删除请求”的留痕表达：不产生替代 request）
- 撤回 response（留痕）：将已具备 response 的 request 标记为作废（voided，保留原 response），并创建新的替代 request 重新发起

## update-task action 的 payload 约定（摘要）
> 说明：`WorkflowResponse.payload` 类型为 `Map<String,Object>`，可直接承载列表/对象；字段结构固定。
- `addRequests`：新增请求列表（数组，元素为 target 字符串）
- `voidRequests`：作废请求列表（数组，元素结构：`{requestId, reason}`）

## 撤回响应（留痕）字段
当撤回某条已响应 request：
- 原 request：`status=voided`，并写入 `voidInfo{voidedAt,reason,by,newRequestId}`，保留原 response 不变
- 新 request：新建 ObjectId，`status=pending`，按正常流程发送

## 留痕（WorkflowEvent）
update-task action 应创建一条人可读的 WorkflowEvent，用于审计时间线，至少包含：
- action 名称（如 ADD_SIGN/WITHDRAW）
- 影响的 requestId 列表（新增/删除/作废）
- 替代关系（oldRequestId -> newRequestId）

## Webhook 影响
- update-task action 应触发 `task.updated` webhook
- 若更新导致新增 request 并成功发送，应继续触发 `request.sent`
- 若产生/撤回 response，应触发 `response.received` 或由实现定义相应的更新通知（当前仍以 `task.updated` 为主）

## 幂等与并发（已采纳）
- 幂等：WorkflowResponse 建议携带 `id(ObjectId)`；引擎对同一 `(taskId, response.id)` 应保证"重复提交不重复生效"
- 并发：对 task 的 update-task 操作应采用乐观锁/版本控制（例如基于内部版本号或数据库层面的并发控制），避免并发更新相互覆盖

## 复合操作约束（新增，已确认）
- 决策类 action 与非决策类 action 不混用：不允许“同一条 response 既用于决策统计又用于更新 task”
- 若产品需要“一键同意并加签”等复合操作，必须拆成两步提交：
  1) 先提交 update-task response（新增/作废 requests）
  2) 再提交决策类 response（参与 emitterRules 产出事件）
