# ADR-010：在配置域引入 state.emitter + state.emitterRules（规则链）以聚合 response 并触发内部事件

## 背景与问题
Task 在 `InProgress` 后可能发出多个 request，每个 request 可能有一个 response。不同场景下：
- 有的希望“任一 response 到达就推进”
- 有的希望“收齐 response 再推进”
- 还有的需要按业务规则/幂等控制决定是否推进

如果把这类策略硬编码进 WorkflowRun 主循环，会使核心循环变复杂且难以扩展。

## 决策
在配置域为每个状态（`WorkflowState`）增加：
- `emitter`：用于声明对外可执行操作清单（actions，含 kind）
- `emitterRules`：规则标识列表（有序），用于聚合 response 并决定触发内部事件

运行时根据该 `state.emitterRules` 按顺序执行规则链来判断：
- 是否触发内部事件（用于定位 `transition.event`）
- 触发哪个内部事件（`eventName` 由 emitter 脚本输出，**不要求等于** `response.action`）

同时约定：
- **每次收到一个 response，都会触发一次 emit**（一次 response -> 一次 emitter 执行）
- emitter 可以选择“不产生事件”（例如需要等待更多 response 汇总后才推进）

## 决策原因
1. **降低主循环复杂度**：把“是否推进”的策略从核心循环中解耦出来。
2. **可扩展性**：支持不同推进策略（收齐/任一/按规则）而不改变核心模型。
3. **便于测试与治理**：推进策略可独立测试与演进（例如幂等、去重、超时策略）。

## 影响与后果（待细化）
- 需要定义 emitter 可访问的上下文（run/task 参数、已收到的 response 集合等）。
- 需要定义 response 去重与幂等策略（同一 request 多次 response 的处理）。
